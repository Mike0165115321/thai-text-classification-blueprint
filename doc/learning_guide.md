# คู่มือการเรียนรู้และสถาปัตยกรรมระบบ: โมเดลแยกแยะข้อความภาษาไทย (Thai Text Classification)

## 1. วัตถุประสงค์และภาพรวมของโปรเจกต์ (Project Overview & Purpose)
**โปรเจกต์นี้สร้างมาทำไม?**
โปรเจกต์นี้ถูกพัฒนาขึ้นเพื่อเป็น **พิมพ์เขียว (Blueprint)** ทางการศึกษา สำหรับผู้ที่ต้องการเรียนรู้และลงมือสร้างระบบ AI แยกแยะหมวดหมู่ข้อความภาษาไทย โดยมุ่งเน้นแสดงให้เห็นถึงความแตกต่างระหว่างการใช้เทคนิคดั้งเดิม (Classical Machine Learning) และการใช้โมเดลระดับ State-of-the-Art อย่าง **WangchanBERTa** (Transformer)

**เป้าหมายหลัก** คือการพิสูจน์ว่า เราสามารถเทรนโมเดล Deep Learning ขั้นสูงที่ซับซ้อน ให้สามารถทำงานได้จริงบนข้อจำกัดของ **คอมพิวเตอร์ทั่วไปที่มี VRAM เพียง 8GB** (เช่น การ์ดจอ RTX 4060) ได้ ผ่านการใช้เทคนิคการปรับแต่ง (Optimization) ขั้นสูง

---

## 2. โครงสร้างและสถาปัตยกรรมของระบบ (System Architecture & Design Principles)
โปรเจกต์นี้ถูกออกแบบโดยยึดหลัก **Separation of Concerns (SoC)** อย่างเคร่งครัด แทนที่จะเขียนทุกอย่างรวมกันในไฟล์เดียว ระบบถูกจัดระเบียบและแยกหน้าที่ออกเป็นส่วน ๆ ดังนี้:

1. **ระดับจัดการข้อมูล (Data Layer):** มีทั้งแบบข้อมูลจำลอง (`dataset.csv`) และการดึงข้อมูลจริงจากระบบคลาวด์ 
2. **ระดับกำหนดบรรทัดฐาน (Baseline Layer):** การใช้แบบจำลองสถิติเชิงเส้นเพื่อหาระดับความแม่นยำขั้นต่ำ (`train_baseline.py`)
3. **ระดับการเรียนรู้เชิงลึก (Training & Optimization Layer):** ส่วนหลักในการสอนโมเดล Transformer ซึ่งถูกแบ่งเป็นขั้นทดสอบลอจิก (`train_bert.py`) และขั้นลงสนามจริงที่จัดการหน่วยความจำ VRAM อย่างเข้มข้น (`train_real_data.py`)
4. **ระดับนำไปใช้งาน (Inference Layer):** การนำโมเดลที่ฉลาดแล้ว ไปใช้งานคาดการณ์ข้อความใหม่ ๆ โดยตัดระบบคำนวณน้ำหนักทิ้งไปเพื่อให้ทำงานได้เร็วที่สุด (`predict_bert.py`)

## 3. เจาะลึกการทำงานทีละส่วน (Deep Dive: Module Breakdown)

### 3.1. ข้อมูลจำลองเบื้องต้น (`dataset.csv`)
* **หน้าที่:** เป็นไฟล์ข้อมูลฝึกสอนส่วนตัวขนาดเล็กมาก (มีแค่ 5 บรรทัด)
* **คุณค่าทางการศึกษา:** เอาไว้ใช้ตรวจเช็กภาพรวม (Sanity Check) ว่าโค้ดระบบการอ่านไฟล์และการเทรนของเราทำงานได้จริง โดยที่ยังไม่ต้องเสียเวลาดาวน์โหลดข้อมูลก้อนใหญ่จากอินเทอร์เน็ต

### 3.2. โมเดลบรรทัดฐานสำหรับเปรียบเทียบ (`train_baseline.py`)
* **หน้าที่:** สร้างตัวเปรียบเทียบประสิทธิภาพเริ่มต้น
* **วิธีการทำงาน:**
  1. ใช้ `pythainlp` ในการตัดคำภาษาไทยให้เป็นคำสั้น ๆ
  2. ใช้ `TfidfVectorizer` แปลงความถี่ของคำให้ออกมาเป็นตัวเลข (Feature Extraction)
  3. สอนด้วยสมการคณิตศาสตร์เชิงเส้น `LogisticRegression`
* **ทำไมต้องมีส่วนนี้:** ในฐานะ System Architect เราต้องมีเครื่องมือวัดที่กินแรงเครื่องน้อยที่สุดเสมอ เพื่อใช้ตอบคำถามว่าโมเดลแพง ๆ ใหญ่ ๆ นั้นคุ้มค่าที่จะลงทุนหรือไม่ หากค่าความแม่นยำ (Accuracy) ของ ML ธรรมดาก็เพียงพอ เราก็อาจไม่จำเป็นต้องใช้ GPU ก้อนโต

### 3.3. การอุ่นเครื่องโมเดล Transformer (`train_bert.py`)
* **หน้าที่:** สาธิตการเขียนโค้ดเพื่อคุยกับสถาปัตยกรรม **WangchanBERTa** (`airesearch/wangchanberta-base-att-spm-uncased`) โดยใช้ชุดข้อมูลจำลอง
* **วิธีการทำงาน:**
  * เปลี่ยนจากระบบพจนานุกรม (PyThaiNLP) มาเป็นการตัดคำย่อยแบบโมเดล (Subword Tokenizer)
  * โครงสร้างใช้ `Trainer` API ของ Hugging Face เพื่อจัดการการเทรนแบบลูปให้โดยอัตโนมัติ
* **คุณค่าทางการศึกษา:** สอนโครงสร้างโครงกระดูก (Boilerplate) หลักสำหรับการสร้าง Deep Learning โดยไม่ทำให้ผู้เรียนสับสนกับปริมาณข้อมูลที่มหาศาล

### 3.4. การฝึกสอนระบบสเกลใหญ่ด้วยการทลายข้อจำกัด VRAM (`train_real_data.py`)
* **หน้าที่:** ศูนย์กลางการทำงานจริง ที่สอน AI ด้วยข้อมูลการรีวิว (Wisesight Sentiment)
* **วิธีการทำงาน:** โหลดข้อมูลผ่าน API ของ Hugging Face, จัดการทำความสะอาด ลบหมวดหมู่ที่ไม่เกี่ยวทิ้ง แล้วแปลงค่าบวก (Positive) เป็น 1 และค่าลบ (Negative) เป็น 0
* **ไฮไลต์สำคัญ (Resource Efficiency Engineering):**
  * ไฟล์นี้ถูกออกแบบมาเป็นพิเศษเพื่อให้คอมพิวเตอร์แรม 8GB เทรนโมเดลขนาดใหญ่ได้รอด โดยอาศัยเทคนิค **Mixed Precision** และ **Gradient Accumulation** (รายละเอียดเต็ม ๆ ในหัวข้อที่ 4)

### 3.5. ระบบนำไปใช้งานจริง (Inference Engine: `predict_bert.py`)
* **หน้าที่:** เรียกใช้งานสมองของ AI ที่ทำการบันทึกไว้ในโฟลเดอร์ `./my_thai_model_real`
* **วิธีการทำงาน:** นำข้อความใหม่มาเข้ากระบวนการ Tokenize จากนั้นโยนเข้า Model แล้วแปลผลลัพธ์ที่เป็นความน่าจะเป็น (Logits) กลับมาเป็น 0 หรือ 1 ผ่านฟังก์ชัน `torch.argmax`

## 4. การจัดการทรัพยากรเชิงรุก (Proactive Resource Optimization)
การที่จะทำให้โมเดลขนาดใหญ่รันบน VRAM ขนาด 8GB ได้จำเป็นต้องใช้เทคนิคทางวิศวกรรมเฉพาะทาง (Resource Efficiency Check) ดังนี้:

1. **ลดปัญหากินแรมตอนอัปเดตน้ำหนัก (Gradient Accumulation):**
   * **เครื่องมือ:** `gradient_accumulation_steps=2` และ `per_device_train_batch_size=4`
   * **เหตุผล (Why/How):** ธรรมชาติของ AI ยิ่งป้อนข้อมูลทีละมาก ๆ (Batch Size ใหญ่) ยิ่งฉลาด แต่ VRAM 8GB ใส่เยอะไม่ได้ โปรแกรมจึงปรับ Batch Size ให้เหลือแค่ 4 ยืดอายุ VRAM แต่สั่งให้ระบบเก็บสะสมการคำนวณไว้ก่อนจนครบ 2 รอบ แล้วค่อยอัปเดตน้ำหนักโมเดล (เท่ากับ 4x2 = 8) ทำให้ประสิทธิภาพเท่าเทียมเครื่องสเปกสูง
2. **หั่นทศนิยมลงครึ่งหนึ่ง (Mixed Precision - FP16):**
   * **เครื่องมือ:** `fp16=True`
   * **เหตุผล (Why/How):** ปกติแล้วเซลล์ประสาทเทียมจะคำนวณด้วยทศนิยม 32-bit การเปิด FP16 จะลดความละเอียดทศนิยมลงเหลือ 16-bit ช่วยหั่นการกินพื้นที่ VRAM ลงครึ่งหนึ่ง และการ์ดจอสมัยใหม่จะมี Tensor Cores ที่รองรับ 16-bit ทำให้คำนวณไวขึ้นอย่างก้าวกระโดด
3. **การเข้าโหมดจำศีลการคำนวณย้อนกลับ (Inference Graph Detachment):**
   * **เครื่องมือ:** `torch.no_grad()` และ `model.eval()` ในฝั่ง Inference
   * **เหตุผล (Why/How):** เวลาใช้งานจริง เราไม่ต้องการให้ AI เรียนรู้เพิ่มอีกแล้ว การสั่ง `no_grad` จะปิดการทำงานเครื่องยนต์คำนวณความผิดพลาด (Autograd Engine) ทำให้กราฟหน่วยความจำหายไปทันที ดึง Latency การตอบสนองให้ลดลงเร็วในระดับเสี้ยววินาที

## 5. การขยายขีดความสามารถและข้อควรระวัง (Scalability and Trade-offs)

### การเติบโตและการทำงานต่อ (Future Scalability)
* **ก้าวข้ามสู่ระบบ Microservice:** ไฟล์ `predict_bert.py` นั้นค่อนข้างคลีน สามารถถูกนำไปสร้างเป็น REST API (เช่น FastAPI) แยกอิสระออกมา และรันเป็น Docker Container ไปฝังที่ Cloud Service ได้
* **ถอดเปลี่ยนสมองได้ดั่งใจ (Model Agnostic):** โค้ดนี้ถูกเขียนให้เป็น Generic API ของ `transformers` หมายความว่าในอนาคต หากมีโมเดลภาษาไทยออกใหม่ที่ดีกว่า WangchanBERTa คุณก็แค่เปลี่ยนชื่อตัวแปรรหัส `model_name` ระบบก็จะทำงานต่อได้ทันทีโดยไม่ต้องแก้โค้ดลอจิกหนัก ๆ 

### สิ่งที่ต้องแลกมา (Trade-offs)
1. **เรื่องความเร็วในการเทรนประหยัดแรม:** การใช้ Gradient Accumulation เซฟแรมได้จริง แต่การหน่วงรอบการอัปเดตจะทำให้เวลาในการนั่งรอเทรนเสร็จนานขึ้น
2. **การถูกตัดหางคำ:** มีการตั้งค่ากรอบ `max_length=128` เพื่อเซฟ VRAM หากข้อความรีวิวยาวเกินโควต้านี้ ท้ายประโยคจะถูกตัดทิ้ง (Truncate) อาจทำให้ความหมายพลิกเพี้ยนได้
3. **ความเทอะทะของ Framework:** โครงสร้าง Transformer ต้องพึ่งพา PyTorch ซึ่งมีขนาดการติดตั้งใหญ่มาก (หลาย GB) หาก Project ของคุณประเมินแล้วว่า Logistic Regression ให้ผลเป็นที่น่าพอใจ การเลือกเทคโนโลยีเดิมอาจคุ้มค่าต่อค่าใช้จ่ายในเชิงธุรกิจมากกว่า
