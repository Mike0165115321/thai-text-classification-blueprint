# 📘 ตำราเรียนฉบับเจาะลึก: วิศวกรรมปัญญาประดิษฐ์ประมวลผลภาษาไทย (Thai NLP Engineering Textbook)

เอกสารฉบับนี้ถูกเขียนขึ้นให้เทียบเท่ากับ **"คู่มือระดับมหาวิทยาลัย (Textbook)"** เพื่อเจาะลึกลงไปในแก่นกลางของการทำงานของไลบรารี สถาปัตยกรรม Transformer และการผ่าตัดโมเดล (Model Surgery) ที่เกิดขึ้นในโปรเจกต์แยกแยะข้อความภาษาไทย (Thai Text Classification) ของเรา

## 🏗️ บทที่ 1: กายวิภาคของคลังคำสั่งและไลบรารีเบื้องหลัง (Anatomy of the Libraries)

การที่จะเป็น System Architect ที่เก่งกาจ จำเป็นต้องเข้าใจว่าเครื่องมือแต่ละชิ้นทำงานอย่างไร "ใต้ฝากระโปรง" (Under the hood):

### 1.1 PyTorch (`torch`)
* **บทบาท:** หัวใจหลักในการคณิตศาสตร์ (The Computational Engine)
* **การทำงานเชิงลึก:** PyTorch เป็นไลบรารีที่ใช้เปลี่ยนข้อมูลทุกชนิดให้เป็นโครงสร้างคณิตศาสตร์ที่เรียกว่า **Tensor** (เมทริกซ์หลายมิติ) ความพิเศษคือมันมีระบบ **Autograd (Automatic Differentiation)** ที่คอยจดจำพฤติกรรมการบวกลบคูณหารทั้งหมดที่เกิดขึ้นกับ Tensor เป็นกราฟต้นไม้ (Computational Graph) 
* **ทำไมถึงต้องใช้:** เมื่อนำกราฟนี้มาควบรวมกับแกนประมวลผล CUDA ของแผงวงจร NVIDIA สถาปัตยกรรมนี้จะสามารถคำนวณการหาอนุพันธ์ (Derivative) แบบย้อนกลับ (Backpropagation) ได้เร็วกว่า CPU ปกติหลายร้อยเท่า นี่คือเหตุผลเบื้องหลังว่าทำไม AI ถึงเรียนรู้ข้อผิดพลาดได้เร็ว

### 1.2 Hugging Face `transformers`
* **บทบาท:** อินเทอร์เฟซระดับสูงสำหรับสถาปัตยกรรม Attention
* **การทำงานเชิงลึก:** ปกติการเขียนโครงสร้าง Transformer (Multi-head Attention, Position Embeddings, Layer Normalization) จากศูนย์ใน PyTorch จะเหนื่อยและใช้เวลาเป็นเดือน ไลบรารีนี้ทำการ "ห่อหุ้ม" (Encapsulate) ความซับซ้อนนั้นไว้เบื้องหลังคลาสอย่าง `AutoModel` และ `Trainer` 
* **จุดเด่นทางวิศวกรรม:** มันอนุญาตให้เราโหลดโมเดลระดับร้อยล้านหรือหมื่นล้านพารามิเตอร์มาใช้งานได้ใน 3 บรรทัด โดยตัวมันเองจะไปสกัดน้ำหนัก (State Dictionary) จากบนคลาวด์มาสวมทับโครงสร้างโครงร่าง PyTorch อัตโนมัติ

### 1.3 Hugging Face `datasets`
* **บทบาท:** ผู้พิทักษ์หน่วยความจำระบบ (RAM Memory Manager)
* **การทำงานเชิงลึก:** ข้อมูลอย่าง Wisesight Sentiment อาจมีขนาดหลายร้อย MB ถึง GB หากใช้ `pandas` โหลดตรงๆ มันจะกิน RAM จนเต็ม (OOM) ไลบรารี `datasets` ทำงานบนระบบ **Apache Arrow** ที่ใช้หลักการแบบ Memory-mapped (Zero-copy) หมายความว่ามันอ่านข้อมูลจากฮาร์ดดิสก์แบบ "สตรีมผ่าน" โดยไม่ต้องดึงข้อมูลทั้งหมดขึ้น RAM ถือเป็นสถาปัตยกรรมระดับ Big Data

### 1.4 PyThaiNLP และ Scikit-learn (สถาปัตยกรรมยุคคลาสสิก)
* **PyThaiNLP (อัลกอริทึม 'newmm'):** เป็นอัลกอริทึมจับคู่คำยาวที่สุดที่ค้นเจอในพจนานุกรม (Dictionary-based Maximum Matching) ซึ่งเป็นจุดอ่อนคลาสสิกของ NLP ไทยยุคเก่า เพราะแปลว่าถ้าเจอ "คำศัพท์แสลงใหม่" ที่ไม่มีในพจนานุกรม มันจะตัดคำพลาด 
* **Scikit-learn (TF-IDF):** ทำงานเชิงสถิติล้วนๆ โดยสมมติฐานตื้นๆ ว่า "คำไหนเจอบ่อยในบทความนี้ แต่ไม่ค่อยเจอในบทความอื่น คำนั้นคือคีย์เวิร์ดสำคัญ" ข้อเสียขั้นร้ายแรงคือ **มันไม่เข้าใจไวยากรณ์และความหมายเชิงลึก (No Semantic Understanding)** ค่อนขอดเรื่องบริบทแวดล้อมไม่ได้

## 🧠 บทที่ 2: ผ่าตัดเปลี่ยนสมองโมเดล - จาก "ผู้รอบรู้ทั่วไป" สู่ "ผู้เชี่ยวชาญเฉพาะทาง"

หัวใจของโปรเจกต์นี้คือกระบวนการ **Transfer Learning** หรือการนำ "ผู้มีความรู้" มาสวมวิญญาณเป็น "ผู้จับผิดอารมณ์" เราจะเจาะลึกความแตกต่างแบบบิตต่อบิต 

### 2.1 โมเดลต้นแบบก่อนขัดเกลา (The Base Model): `wangchanberta-base-att-spm-uncased`
* **ตัวตนที่แท้จริง:** เป็นโมเดลสถาปัตยกรรม RoBERTa (Transformer ฝั่ง Encoder-only พัฒนาต่อจาก BERT ของ Google) ซึ่งถูกสร้างโดยศูนย์วิจัย VISTEC และ AIResearch ของไทย
* **น้ำหนัก (Parameters):** 110 ล้านตัวแปรคณิตศาสตร์
* **แหล่งตั้งต้นประจุรอยหยัก (Pre-training Data):** ถูกส่งเข้าไปดมคลังข้อความภาษาไทยที่มีขนาดมหึมาถึง **78.5 GB** (กวาดตั้งแต่เนื้อหากระทู้ Pantip, Wikipedia, Twitter ไปจนถึงข่าว)
* **สถานะความตระหนักรู้ (Capability):** มันเข้าใจไวยากรณ์ไทย เข้าใจภาษาวัยรุ่น โครงสร้างประโยค และความสัมพันธ์ของประธาน-กริยา-กรรม ได้ดีเลิศ **ทว่า... มันยังคงแยกแยะความรู้สึกบวก/ลบไม่เป็น** เพราะในอดีตมันถูกสอนมาแค่เกมปริศนาทายคำ "ให้เดาคำที่โดนเซ็นเซอร์ไป (Masked Language Modeling)" เท่านั้น 
* **กลไกตัดคำผ่านความน่าจะเป็น (SentencePiece):** ไม่ใช้พจนานุกรมหยาบๆ แบบ PyThaiNLP แต่มันใช้ระบบสถิติย่อย (Subword Tokenization แบบ Unigram) เช่น เจอคำว่า "โคตรปัง" ยุคนั้นไม่มีในดิกชันนารี มันจะหั่นสมองเป็น "โคตร" + "ปัง" ทำให้มัน **รอดพ้นจากอาการ Out-of-Vocabulary (OOV)** 

### 2.2 โมเดลหลังผ่านการขัดเกลา (The Fine-Tuned Model): `my_thai_model_real`
* **ตัวตนที่แท้จริง:** ศิษย์เอกที่เปลี่ยนจากการเรียนทั่วไป (General) มาเรียนสายวิชาชีพ (Specialized Task) 
* **กระบวนการศัลยกรรมประกอบร่าง (Model Surgery):** ทันทีที่เราเรียกใช้คลาส `AutoModelForSequenceClassification` โค้ดจะคีมตัด "ส่วนหัวทายคำ" (Classification Head) ของ WangchanBERTa ทิ้งไปลงถังขยะ แล้วบรรจงสร้างโครงข่ายประสาทชุดใหม่เอี่ยมขนาดเล็ก (เป็น Neural Network ชั้นเดียวที่มีปลายทางแค่ 2 โหนด คือ Positive = 1, Negative = 0) สวมทับเข้าไปที่คอของโมเดลเดิม
* **การอุดรอยรั่วและซึมซับความรู้:** ตอนที่มันเริ่มอ่านรีวิวจาก `wisesight_sentiment` ไหลผ่านกระบวนการ Backpropagation โมเดลจะพยายามปกป้องน้ำหนักความเข้าใจภาษาไทยชุดดั้งเดิมไว้ (ป้องกัน Catastrophic Forgetting) แต่ในขณะเดียวกัน ก็มุ่งปรับจูนน้ำหนักและสมการเฉพาะบริเวณ "หัวใหม่เอี่ยม" และเลเยอร์ลึกสุด ให้เปิดเรดาร์เซนซิทีฟกับบริบทเชิงอารมณ์ คัดแยกน้ำพักน้ำแรงของประโยคด่านั้นๆ
* **ผลลัพธ์ที่ตกผลึก:** โมเดลที่มีพารามิเตอร์ 110 ล้านตัวเท่าเดิมเป๊ะ แต่น้ำหนักคณิตศาสตร์ถูกดัด (Biased) ให้เอนเอียงเข้ากับการทำ Sentiment Analysis เฉียบคมระดับมนุษย์

### 📊 สรุปตารางเปรียบเทียบ (Base vs Fine-Tuned)

| คุณสมบัติทางสถาปัตยกรรม | Base Model (`wangchanberta-base-att-spm-uncased`) | Fine-Tuned Model (`my_thai_model_real` ที่ฝึกเสร็จ) |
| :--- | :--- | :--- |
| **เป้าหมายสถาปัตยกรรมเลเยอร์ท้าย** | คาดเดาคำศัพท์ที่โดนหน้ากากปิดบัง (Masked Text Prediction) | ทายหมวดหมู่อารมณ์บวก/ลบ (Sequence Classification) |
| **ขนาดไฟล์โมเดล (Disk Size)** | ~ 400 MB | ~ 400 MB (ขนาดคงที่ แค่ตัวเลขน้ำหนักในเมทริกซ์เปลี่ยนแปลง) |
| **ความรู้ภาษาไทยพื้นฐาน** | ปริญญาเอกภาษาไทย (ซึมซับจาก Text 78.5 GB) | คงระดับปริญญาเอกไว้ได้ครบถ้วน (Transfer Learning Rule) |
| **ความเข้าใจทางอารมณ์/ความรู้สึก** | ไม่เข้าใจ (หากให้ตอบ บังคับสุ่มมั่วร้อยเปอร์เซ็นต์) | เชี่ยวชาญระดับนักวิเคราะห์ข้อมูลรีวิว (Domain Specialist) |
| **ค่าใช้จ่ายฮาร์ดแวร์ในการเทรน** | ต้องใช้ Supercomputer เช่าหลายหมื่นดอลลาร์รันเป็นเดือน | ใช้คอมทั่วไป (RTX 4060) รันบนฝั่งเราจบในไม่กี่ชั่วโมง |

## ⚡ บทที่ 3: ความลับเจาะทะลุสถาปัตยกรรมคอขวด (Hardware Bottlenecks & Optimization)

การเทรน AI ไม่ใช่เวทมนตร์ แต่เป็นวิชาแคลคูลัสที่ผูกมัดติดกับหน้าตักของฮาร์ดแวร์ (VRAM 8GB) นี่คือวิศวกรรมการเอาชนะคอขวดเหล่านั้น

### 3.1 วิกฤติกระทะทองแดงหน่วยความจำบวม (The VRAM Explosion Crisis)
การ์ดจอ VRAM 8GB แทบจะเป็นลูกเมียน้อยของวงการ Deep Learning เพราะเวลาสอน AI มันไม่ได้ทำแค่เอาโมเดล 400 MB ไปวางไว้:
1. การเก็บสถานะ Optimizer (เช่น AdamW) ตัวมันต้องจดจำทิศทางความชัน (Momentum) เผื่อไว้ดึงรั้งตอนลงเขา ซึ่งกินพื้นที่อีกประมาณ 3-4 เท่าของตัวโมเดล (พุ่งทะลุไป 1.3 GB ทันที)
2. วิกฤติที่แท้จริงคือ **Activations Memory** มันคือเศษซากผลคูณเมทริกซ์ตลอดยาวทางเดิน (Forward Pass) ที่ระบบต้องอั้น/กลั้นเอาไว้ไม่ยอมลบทิ้ง เพื่อเฝ้ารอนำไปใช้ตีกลับตอนทำ Backpropagation ยิ่งคุณตั้ง Batch Size ใหญ่ หรือใส่ Length ประโยคยาวสุดโต่ง (เกิน 128 Token) กราฟ Activation ก้อนนี้จะพองโตเป็นลูกโป่ง จนชนเพดาน 8GB และแจก Error สยองขวัญ `CUDA Out of Memory` ให้คุณชื่นใจ

### 3.2 กายวิภาคลดทรานซิสเตอร์ด้วย FP16 (Mixed Precision Engineering)
เมื่อเราใส่คำสั่งลับ `fp16=True` เป็นการบังคับให้สเปกคณิตศาสตร์บนการ์ดจอ เปลี่ยนจากการจับถือตัวเลขทศนิยมที่มีความละเอียดโคตรสูง (32-bit: `1.12345678`) ให้เหลือแค่หยาบๆ (16-bit: `1.1234`) 
* **สิ่งที่ได้กลับมา:** น้ำหนักโมเดล กราฟซาก Activation ทั้งหมดของระบบ จะถูกสับหั่นครึ่งให้ล่องหนหายไปทันที (Memory footprint ลดฮวบไป 50%) 
* **อาวุธระดับแสง:** โค้ดตัวนี้จะปลุก **Tensor Cores** (ชิปเซตพิเศษในตระกูลสถาปัตยกรรม Turing/Ampere/Ada ของ NVIDIA ที่ถูกสร้างมาคูณเมทริกซ์ 16-bit แบบเฉพาะกิจ) มาดึงกระบวนการประมวลผลแทน Cuda Core สายพานแบบดั้งเดิม ส่งผลให้ความเร็วคำนวณ (TFLOPS) พุ่งทะยานขึ้นหลายเท่าตัว

### 3.3 การตัดต่อพันธุกรรมสร้างแคชสะสม (Gradient Accumulation)
มีกฎเหล็กว่า: *"AI ของคุณจะเข้าใกล้จุดเสถียรสุดยอดได้ ก็ต่อเมื่อมันกวาดตาดูก้อนข้อมูลรวมเป็นหลักสิบ/หลักร้อยในเสี้ยววิ (Large Batch Size)"* แต่ VRAM ตัวกระจ้อยร่อยระดับ 8GB อนุญาตให้ทำแบบนั้นไม่ได้ ฮาร์ดแวร์บีบคอให้เราตั้งค่า `per_device_train_batch_size=4` ซึ่งเล็กเป็นมด ส่งผลให้ฉลาดช้าและแกว่งไปมาเวลาเรียนรู้
* **สถาปัตยกรรมการแฮก (Hack Layout):** `gradient_accumulation_steps=2` 
* **กลไกการทำงานลึกซึ้ง (How it works):** แทนที่พอวนลูปข้อมูลชุดละ 4 รูปเสร็จปุ๊บ จะรีบเร่งสั่งให้ AI อัปเดตปรับความทรงจำทันที (Optimizer Step) ระบบแสร้งทำเป็นใจเย็น แอบเก็บรวบรวมค่ากิเลสความผิดพลาดระดับไมโคร (Gradient Logs) นั้นสะสมไว้ในลิ้นชักชั่วคราว แล้วหยิบข้อมูลล็อตต่อไปอีก 4 รูปมาวิ่งผ่านระบบ เมื่อทบยอดจัดสรรครบ 2 ล็อต (4+4 = เสมือนมีพลัง VRAM รัน Batch Size ปลอมได้ถึง 8) มันค่อยส่งสัญญาณจุดพลุให้ Optimizer ดัดแปลงโมเดลรวดเดียว! ผลลัพธ์คือการสกัดประสิทธิภาพขั้นสูงสุดภายใต้ทรัพยากรกิ๊กก๊อก แลกมาด้วย Cycle เวลาคำนวณของ CPU/RAM เล็กน้อย

### 3.4 โหมดจำศีลสุญญากาศของ Inference (`predict_bert.py`)
เมื่อคุณเทรนเสร็จ ความทรงจำโมเดลถูกแช่แข็งแล้ว ในโค้ด `predict_bert.py` ผมได้ลั่นดาลคำสั่งป้องกันกัลปาวสานฮาร์ดแวร์ไว้ให้เรียบร้อย:
1. **`model.eval()`**: การอ้าปากตะโกนลงกราฟของทุกๆ เลเยอร์ว่า "ช่วงเวลาแห่งการเรียนจบสิ้นแล้ว" ฟังก์ชั่นประหลาดอย่าง Dropout (การสุ่มยิงดับเซลล์สมองให้ลืมคำตอบเก่า เพื่อบังคับให้มันวิเคราะหาสิ่งใหม่) จะถูกหยุดทำงานทันที นี่คือตัวการ์รันตีความถูกต้องของผลพยากรณ์ให้เสถียรแตะเพดาน 100%
2. **`with torch.no_grad():`**: นี่คือหมัดน็อกการกินทรัพยากร! เราสั่งประหารระบบ AutoGrad Engine ทิ้งไปเลย แปลว่าระบบกราฟจดจำความผิดพลาดย้อนหลังที่จะสูบแรมมหาศาลจมเขี้ยว จะไม่ถูกวาด/สร้างขึ้นมาอีกแล้ว ทำให้ความสยองของ Memory ลดฮวบ และทำให้อัตราความหน่วงเวลาตอบกลับ (Inference Latency) วิ่งฉลุยในระดับเสี้ยว 100 Milliseconds จนสามารถนำไปวางขึ้นเป็น Real-Time Application ได้สบายๆ

## 🔮 บทที่ 5: จาก Prototype สู่ระดับโลก (Production Readiness & Trade-offs)

และสุดท้ายนี่คือสัจธรรมที่ Software Architect ทุกคนต้องเข้าใจ:

**1. ขวดแก้วจำกัดบริบท (Sequence Bottleneck):**
เราบีบ `max_length=128` มันคือกรอบการมอง (Receptive Field) หากนำมาส่องทวีต, คอมเมนต์ทั่วไป มันคมกริบ แต่ถ้านำไปวิเคราะห์โพสต์บ่นยาวเป็นนิยาย 5 ย่อหน้า ระบบจะคว้ามาอ่านแค่อารัมภบท 128 คำแรกแล้วเฉือนหางทิ้ง (Truncate) นี่อาจสร้างความเสี่ยงกรณีที่ผู้ใช้งาน "เขียนด่ารัวๆ ตอนต้น แต่มาหักมุมชมยับๆ ตอนท้ายประโยค"

**2. การยกระดับพ้นเงา PyTorch (WebAssembly / ONNX Runtime):**
ถ้าความแม่นยำจุดนี้ของคุณ Perfect แล้ว สถาปัตยกรรมระดับ Enterprise คือต้องกะเทาะเปลือก PyTorch ที่หนักเป็น Gigabyte ทิ้งไป แล้วเข้าสู่กระบวนการ **ONNX Serialization** หรือการส่งออกน้ำหนักโมเดลให้กลายเป็นแค่โครงข่ายลอจิกเพียวๆ การทำแบบนี้จะทำให้โปรเจกต์คุณเบาหวิวดั่งฝุ่น ผันตัวไปสิงสถิตอยู่ในเบราว์เซอร์, รันบนมือถือ natively, หรือเขียน C++, Rust ครอบเป็น Edge API ที่ดึงประสิทธิภาพสูงสุดด้วยฮาร์ดแวร์ระดับไมโครได้สบายๆ
